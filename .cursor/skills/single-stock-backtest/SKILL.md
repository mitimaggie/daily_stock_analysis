# A 股单股快速回测

快速验证代码改动是否走通的端到端测试流程。仅适用于 A 股。

## 使用场景

- QA 质检员被召唤后的第三层测试
- 代码修改后的冒烟测试
- 新功能开发完成后的基础验证

## 测试流程

### 第一步：确认环境

```bash
test -f .env && echo "ENV OK" || echo "缺少 .env 文件"
```

### 第二步：选择测试股票

默认使用以下高流动性 A 股进行测试（按优先级）：
1. `600519` - 贵州茅台（沪市大盘蓝筹，数据最稳定）
2. `000001` - 平安银行（深市主板，备选）
3. `300750` - 宁德时代（创业板，验证 20% 涨跌停处理）

### 第三步：运行单股分析

```bash
# 标准单股测试（推荐）
python main.py --stocks 600519 --no-notify --debug

# 快速模式（跳过外部搜索，用缓存，适合纯逻辑验证）
python main.py --stocks 600519 --no-notify --debug --fast

# 仅验证数据抓取（dry-run 模式）
python main.py --stocks 600519 --dry-run --debug
```

### 第四步：检查输出

重点验证以下关键节点：
1. **数据抓取**：日志中出现 "获取数据成功" 或类似正常返回信息
2. **指标计算**：没有 NaN、Inf 或除零错误
3. **LLM API 调用**：返回了有效的 JSON 响应，没有 429/500 错误
4. **输出格式**：最终分析报告结构完整
5. **散户合理性**：输出的建议是否是散户可执行的

### 第五步（可选）：边界场景测试

如果改动涉及数据处理或指标计算，建议额外测试：

```bash
# 测试创业板股票（20% 涨跌停）
python main.py --stocks 300750 --no-notify --debug --fast
```

### 异常处理

| 错误类型 | 可能原因 | 快速排查 |
|---------|---------|---------|
| `ConnectionError` | 网络/代理问题 | 检查 USE_PROXY 和代理设置 |
| `APIError 429` | Gemini 限流 | 等待 60s 重试，或检查 gemini_request_delay |
| `KeyError` | A 股数据源字段变更 | 检查 data_provider 返回的字段名 |
| `JSONDecodeError` | LLM 输出格式异常 | 检查 Prompt 是否要求了 JSON 输出 |
| `ValueError: NaN` | 停牌/缺失数据未处理 | 检查停牌日和数据空值兜底逻辑 |
