# 《量化与算法优化方案》——持仓监控实时行情缓存 TTL 参数评估

## 一、实时行情缓存 TTL（`_realtime_cache`）

### 1.1 当前状态

| 项目 | 值 |
|------|-----|
| 代码位置 | `data_provider/akshare_fetcher.py` 第 35 行 |
| 当前 TTL | **1200 秒（20 分钟）**（硬编码，未使用 config） |
| Config 定义 | `config.realtime_cache_ttl = 600`、`config.cache_ttl_realtime_quote = 600`（均未被 akshare_fetcher 读取） |
| 数据源 | `ak.stock_zh_a_spot_em()`，单次返回全市场 5000+ 股票 |
| 共享方 | akshare_fetcher（单股行情）、market_sentiment（涨跌家数） |

### 1.2 根因分析

- **业务矛盾**：前端每 2 分钟轮询，但缓存 20 分钟有效 → 连续 10 次刷新看到同一价格，止损信号时效性严重不足。
- **A 股盘中价格变化**：9:30–15:00 内价格每秒变动，2–5 分钟内可能出现明显波动（如 -3% ~ -5%），20 分钟 stale 数据对止损决策不可接受。
- **akshare 限流**：`stock_zh_a_spot_em` 为东财全市场接口，反爬严格；项目已有 rate_limiter + 2–5 秒请求间隔，需在新鲜度与封禁风险间折中。

### 1.3 推荐方案

**推荐：盘中 120 秒，盘后 1800 秒（区分盘中/非盘中）**

| 维度 | 盘中（9:30–15:00） | 盘后/非交易日 |
|------|---------------------|---------------|
| TTL | **120 秒（2 分钟）** | **1800 秒（30 分钟）** |
| 理由 | 与前端轮询间隔对齐，每次轮询有机会拿到新数据 | 收盘后行情不变，可大幅延长 TTL 降低请求 |

**量化对比**：

| 维度 | 优化前 | 优化后（盘中） | 优化后（盘后） |
|------|--------|----------------|----------------|
| TTL | 1200s | 120s | 1800s |
| 每 2 分钟轮询命中新数据概率 | 0% | 100% | N/A（盘后不轮询） |
| stock_zh_a_spot_em 调用频率（盘中） | 3 次/小时 | 30 次/小时 | - |
| 限流风险 | 低 | 中等（需观察） | 极低 |

**限流风险评估**：

- 30 次/小时 ≈ 每 2 分钟 1 次，与 rate_limiter 的 2–5 秒间隔兼容。
- 项目内参考：资金流向盘中 180s、市场情绪盘中 300s；全市场接口略重，120s 略激进，**若出现封禁可退至 180s（20 次/小时）**。

**备选方案**（若 120s 触发限流）：

- **盘中 180 秒（3 分钟）**：20 次/小时，每 3 次轮询约 2 次命中新数据，止损时效仍可接受。

## 二、分时分析缓存 TTL（`_CACHE_TTL`）

### 2.1 当前状态

| 项目 | 值 |
|------|-----|
| 代码位置 | `data_provider/intraday_fetcher.py` 第 29 行 |
| 当前 TTL | **300 秒（5 分钟）** |
| 数据源 | `ak.stock_zh_a_hist_min_em(symbol=code)`，**按股票单独请求** |
| 用途 | VWAP、分时趋势、量能分布等，供持仓监控、Flash 预判、盘中分析 |

### 2.2 根因分析

- 分时信号（VWAP 位置、分时上攻/下跌/震荡）基于分钟 K 线，变化频率低于 tick 级价格。
- 5 分钟级别数据本身有滞后，1–2 分钟内的微小波动对「分时趋势」判断影响有限。
- 接口为**单股**请求，与全市场 `stock_zh_a_spot_em` 不同，不会因单次请求影响全市场缓存。

### 2.3 推荐方案

**推荐：保持 300 秒（5 分钟），或盘中 300s / 盘后 600s**

| 维度 | 建议 |
|------|------|
| 盘中 TTL | **300 秒（5 分钟）**，维持现状 |
| 盘后 TTL | **600 秒（10 分钟）**（可选，非交易时段分时无意义，可适当延长） |

**理由**：

- 分时趋势、VWAP 等指标在 5 分钟内变化有限，300s 已足够。
- 持仓监控每 2 分钟轮询，分时数据 5 分钟更新一次，与「价格每 2 分钟更新」形成层次：价格更新更频繁，分时略慢，符合业务逻辑。
- 单股接口调用压力小于全市场接口，300s 不会显著增加限流风险。

**复杂度对比**：无实质变化，保持现状即可。

## 三、是否区分盘中/非盘中 TTL

### 3.1 结论：**建议区分**

| 时段 | 数据特性 | 推荐策略 |
|------|----------|----------|
| 盘中（9:30–15:00） | 价格、分时持续变化 | 短 TTL，保证止损与信号时效 |
| 盘后（15:00 后、午休 11:30–13:00、非交易日） | 行情冻结 | 长 TTL，减少无效请求 |

### 3.2 实现思路（伪代码）

```python
# akshare_fetcher._get_em_quote 内
def _get_realtime_ttl() -> int:
    if _is_market_open():
        return 120   # 盘中 2 分钟
    return 1800      # 盘后 30 分钟

# 检查缓存时使用动态 TTL
ttl = _get_realtime_ttl()
if current_time - _realtime_cache['timestamp'] < ttl:
    df = _realtime_cache['data']
else:
    # 重新拉取
```

**参考**：`market_sentiment` 已采用 `l1_ttl = 300 if _is_market_open() else 1800`，资金流向采用 `cache_ttl_capital_flow_intraday=180` vs `cache_ttl_capital_flow=600`，模式一致。

## 四、参数汇总与实施建议

| 缓存类型 | 盘中 TTL | 盘后 TTL | 是否区分 | 备注 |
|----------|----------|----------|----------|------|
| 实时行情 `_realtime_cache` | **120s** | **1800s** | 是 | 与前端 2 分钟轮询对齐；若限流可改为 180s |
| 分时分析 `_CACHE_TTL` | **300s** | 300s 或 600s | 可选 | 当前 300s 可接受，盘后区分非必须 |

**实施建议**：

1. 将 `_realtime_cache['ttl']` 改为从 `_is_market_open()` 动态计算，或接入 `config.realtime_cache_ttl` 并新增 `config.realtime_cache_ttl_after_hours`。
2. 上线后观察 akshare 是否出现封禁或 429，若有则把盘中 TTL 从 120s 调至 180s。
3. 分时缓存保持 300s，暂不区分盘中/盘后；若后续有盘后分时需求，再增加盘后 600s。
