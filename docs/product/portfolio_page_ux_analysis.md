# 《持仓页体验改进方案》—— 产品分析报告

**分析人**：PM  
**触发原因**：老板反馈持仓页使用"卡手"，以及放大镜、概念热度两个具体问题  
**分析日期**：2026-03-10

---

## 一、问题一：放大镜按钮行为不符预期

### 1.1 用户场景

散户在持仓页看到某只股票，想深入看分析报告，点击放大镜 → **期望一键直达该股票的分析报告**。

### 1.2 当前实现与断裂点

| 环节 | 当前行为 | 用户期望 |
|------|----------|----------|
| 点击放大镜 | `navigate('/analysis?stock=600519')` | 同上 |
| 分析页加载 | **不读取 URL 的 `stock` 参数** | 自动填充代码并触发分析 |
| 用户看到 | 空的分析页或最近一条历史记录 | 该股票的报告（有缓存直接显示，无缓存自动开始分析） |

**根因**：`HomePage`（分析页）未使用 `useSearchParams` 或 `location.search` 读取 `stock` 参数，跳转后完全忽略该参数。

### 1.3 建议方案

**方案**：跳转时带上股票代码参数，分析页加载时：
1. 读取 URL 参数 `stock`（或兼容 `code`）
2. 若有参数：填充到搜索框 → 先查历史是否有该股票报告 → 有则直接展示，无则**自动点击「分析」触发分析**
3. 清除 URL 参数（避免刷新时重复触发）

**涉及角色**：Frontend（HomePage 增加 URL 参数读取 + 自动触发逻辑）

**优先级**：P0（直接解决"卡手"的核心断裂点）

## 二、问题二：市场概念热度模块价值不清

### 2.1 模块原始意图

帮散户判断**持仓是否踩中热门概念**，辅助加减仓决策。例如：持仓里有 AI 概念股，而 AI 概念今日涨幅 Top 3，可增强持有信心；反之若持仓概念都在降温，可考虑减仓。

### 2.2 为什么当前实现没达到目的

- 展示的是「市场概念热度 Top 10」列表，每个概念有：名称、涨跌幅、排名、龙头股
- **缺少持仓股与概念的关联标注**：散户看完后不知道「我的哪只股票属于哪个概念」
- 文案「对照你的 3 只持仓，看看哪些概念正在发力」制造了期待，但列表没有兑现

### 2.3 改进方案对比

| 方案 | 描述 | 价值 | 成本 | 推荐 |
|------|------|------|------|------|
| **A** | 在每个概念后面标注「你的 XXX 属于该概念」 | 中 | 低（需后端返回概念-持仓映射） | ✓ 优先 |
| **B** | 以持仓为主体：「你的持仓关联的热门概念」 | 高 | 中（需重新设计模块结构） | ✓ 可选升级 |
| **C** | 若持仓与 Top 10 无交集，明确告知「你的持仓未涉及当前热门概念」 | 中 | 低 | ✓ 必须 |
| **D** | 去掉该模块 | 无 | 无 | ✗ 不推荐 |

### 2.4 推荐实现路径

1. **第一步（A+C）**：  
   - 后端：扩展 market overview 或新增接口，返回「Top 10 概念 + 每个概念下属于用户持仓的股票列表」  
   - 前端：在每个概念行后展示「你的 XXX、YYY 属于该概念」；若某概念下无持仓股，不展示；若 Top 10 全部无持仓，展示「你的持仓未涉及当前热门概念」
2. **第二步（可选 B）**：以持仓为主体，展示「每只持仓股所属的热门概念及热度」，更适合「先看持仓再看概念」的心智

**数据基础**：后端已有 `stock_concept_mapping` 表和 `get_stock_concepts(code)`，可复用。

**涉及角色**：DataEngineer（确认映射数据可用性）→ Coder（API）→ Frontend（展示）

**优先级**：P1（提升模块价值，减少"看不懂跟我有什么关系"的困惑）

## 三、问题三：整体"卡手"感 —— 体验障碍分析

### 3.1 信息密度

| 问题 | 现状 | 影响 |
|------|------|------|
| 首屏信息过多 | 汇总卡片（总资金/持仓市值/空仓/浮盈亏/今日盈亏/风险信号）+ 权重分布条 + 概念热度 + 集中度预警 + 持仓卡片 | 散户打开 3 秒内难以聚焦「最该看什么」 |
| 卡片内容密集 | 每张持仓卡：信号、成本、现价、浮盈、评分、ATR 止损、锁住浮盈、持仓高点、再分析提醒、分时简报、信号原因、P&L 条、操作日志 | 信息过载，关键信息（浮盈、信号）被稀释 |

**建议**：遵循「三秒法则」—— 首屏最显眼处应是「总浮盈亏 + 风险信号数量」，其余可折叠或弱化；持仓卡片可考虑「摘要行 + 展开详情」模式。

### 3.2 操作动线

| 用户目标 | 当前路径 | 断裂点 |
|----------|----------|--------|
| 查看盈亏 | 打开持仓页 → 看汇总卡片 | ✓ 基本顺畅 |
| 看某只股票分析 | 点放大镜 → 分析页 → **需再输入/选择** | ✗ 断裂（见问题一） |
| 做决策（加减仓） | 看信号 → 需去报告页确认 → 再回来操作 | 来回跳转，动线不闭环 |
| 记录操作 | 展开操作日志 → 选动作/填价格/备注 → 记录 | ✓ 可用，但入口较深（需先点 📋） |

**建议**：优先修复放大镜一键直达；其次考虑在持仓卡上增加「快速查看报告」的简化入口（如弹窗或侧边栏），减少页面跳转。

### 3.3 "看不懂"的信息

| 信息 | 散户可能反应 | 建议 |
|------|--------------|------|
| ATR 止损 | 不懂 ATR 是什么 | 增加 tooltip：「基于波动率计算的止损价，跌破建议减仓」 |
| 评分数字（如 72） | 不知道 72 算好还是差 | 已有颜色区分（红/黄/绿），可补充「70+ 偏多」等简短说明 |
| 锁住浮盈 X% | 与「浮盈」的区别不清晰 | tooltip：「若在 ATR 止损价卖出，可锁定的浮盈比例」 |
| 持仓周期意图（短线/中线等） | 相对清晰 | 可保持 |
| 信号标签（止损/减仓/加仓/持有） | 基本清晰 | 可保持 |

### 3.4 "找不到"的功能

| 功能 | 入口 | 可发现性 |
|------|------|----------|
| 操作日志 | 卡片右上角 📋 图标 | 低（无文字，仅 title） |
| 简化视图 | 📊 图标 | 低 |
| AI 分析 | 🔍 图标 | 低 |
| 移除持仓 | × 按钮 | 中 |
| 新增持仓 | 右上角按钮 | 高 |

**建议**：所有图标按钮增加 **tooltip**（已有 title 的需确保文案清晰），或改为「图标+短文字」如「📋 日志」「🔍 分析」，降低认知负担。

### 3.5 其他卡手点

- **关注股 Tab 的「分析」按钮**：当前 `window.location.href = '/?code=XXX'` 跳转到根路径 `/`（MarketPage），而非 `/analysis`。用户点「分析」期望去分析页，这是**路由错误**，需改为 `/analysis?stock=XXX` 并配合问题一的修复。
- **概念热度模块位置**：在汇总卡片与持仓卡片之间，用户滚动时可能忽略；若与持仓关联后价值提升，可考虑保持位置或略微上移。

## 四、改进方案优先级排序

| 优先级 | 改进项 | 预期效果 | 涉及角色 |
|--------|--------|----------|----------|
| **P0** | 放大镜一键直达分析 | 消除「点完还要再操作」的断裂感 | Frontend |
| **P0** | 关注股「分析」按钮跳转修正 | 修复路由错误，与分析页一致 | Frontend |
| **P1** | 概念热度标注持仓归属 | 回答「这跟我有什么关系」 | Coder + Frontend |
| **P1** | 概念热度无交集时明确告知 | 避免用户困惑「是不是坏了」 | Frontend |
| **P2** | 卡片图标增加 tooltip/文字 | 降低「找不到」的挫败感 | Frontend |
| **P2** | ATR/锁住浮盈等增加说明 | 减少「看不懂」的焦虑 | Frontend |
| **P3** | 首屏信息层级优化 | 三秒内聚焦关键信息 | Frontend + PM 交互设计 |
| **P3** | 持仓卡摘要+展开模式 | 降低信息过载 | Frontend |

## 五、总结

老板反馈的「卡手」感，主要来自：

1. **操作断裂**：放大镜和关注股分析按钮未实现「一键直达」，需要多余步骤
2. **价值不清**：概念热度模块缺少与持仓的关联，用户无法建立「我的股票 ↔ 市场热点」的认知
3. **认知负担**：图标无说明、专业术语无解释、信息密度高

建议按 P0 → P1 → P2 → P3 顺序推进，优先解决操作动线断裂，再提升信息可理解性和模块价值。
